Daily collection scope
Collect from GitHub public repositories using official GitHub APIs. Do not scrape GitHub HTML pages unless absolutely necessary. Use API-first design.

Sources to use
1. GitHub Search Repositories API for discovery
2. GitHub Repository detail endpoint for metadata
3. GitHub READMEcontent endpoint for README extraction
4. GitHub Topics if available in repository metadata
5. GitHub Languages endpoint
6. GitHub Releases endpoint, limited
7. GitHub IssuesPulls endpoint, limited and only for aggregate signals
8. GitHub Events API only as optional near-real-time activity signal, not as a true real-time source

Important API behavior to design around
- Treat GitHub Events API as near-real-time only, not true real-time.
- Respect rate limits.
- Read and honor x-ratelimit headers.
- Implement retries and exponential backoff for transient 4034295xx.
- For Events API polling, use ETag and X-Poll-Interval if implemented.
- Avoid unnecessary code search calls because code search has stricter limits.
- Cache responses per run.
- Do not run unbounded collection.
- Make max repos per run configurable.
- Make max details per run configurable.
- The pipeline must remain useful even with limited API budget.

Image generation requirement
The user wants some important and complex content to use meaningful visuals, either realistic images or GPT Image 2 generated images.

Implement optional image generation as follows
- Controlled by ENABLE_IMAGE_GENERATION.
- Default false.
- Use IMAGE_MODEL default gpt-image-2.
- Only generate images for top IMAGE_TOP_N projects or daily hero cards when
  - overall score = threshold, or
  - project is complex and important, or
  - project is selected as daily top story.
- Do not generate images for every repo.
- Save images under
  assetsimagesYYYY-MM-DD{safe_repo_slug}.png
- Save prompt metadata under
  assetsimagesYYYY-MM-DD{safe_repo_slug}.json
- Save daily hero image under
  assetsimagesYYYY-MM-DDdaily-hero.png
- Save OGsocial card under
  assetscardsYYYY-MM-DD-og-card.png

Image generation style
- Realistic editorial  data product  open-source intelligence style
- Clean, professional, portfolio-friendly
- No copyrighted characters
- No fake GitHub UI screenshots
- No real person likeness
- No unauthorized logos
- Do not ask the image model to render exact statistics or long text.
- Generate image backgroundvisual metaphor first.
- Programmatically overlay exact headlinedatescore using Pillow if needed.
- Every generated image must have alt text.
- Label generated images as AI-generated in metadata or caption when appropriate.
- If OpenAI API fails, create a deterministic fallback SVGPNG card with exact text and simple layout.
- The pipeline must still complete if image generation fails.

Important design detail
The daily report should be “report-as-data”. That means every generated Markdown report should correspond to a JSON data file containing the same core records. The dashboard should read structured JSONCSV, not scrape Markdown.

Archive ordering
Use generated_at ISO timestamp from RunMetadata for ordering.
Do not order only by filename if generated_at exists.
docsdataarchive_index.json should include
[
  {
    date YYYY-MM-DD,
    generated_at ...,
    daily_brief_path ...,
    json_path ...,
    csv_path ...,
    selected_count 20,
    top_project ownerrepo
  }
]

Data freshness
For each repo include
- first_seen_date
- last_seen_date
- days_seen
- rank_today
- rank_previous
- rank_change
- star_delta_since_previous_seen if available
- fork_delta_since_previous_seen if available

If no previous snapshot exists
- Compute approximate momentum from starsrepo_age_days and recent pushed_at.
- Mark delta fields as null, not zero.

Security
- Never commit tokens.
- Do not log Authorization headers.
- .env in .gitignore.
- Keep generated metadata safe.
- Treat README content as untrusted text; do not execute code from repos.
- Do not clone arbitrary repos in MVP.
- Do not run repository code.
- Only inspect metadataREADMEfiles via API.
- Do not scrape or download arbitrary images from repos for reuse unless license and source are clear.
- Prefer generated visuals or programmatic cards.

Limitations section
Document
- GitHub API is near-real-time, not guaranteed real-time.
- Search API has result and rate limits.
- Scores are heuristics.
- Star velocity is approximate unless previous snapshots exist.
- Optional LLMimage generation may cost money and require API key.
- The dashboard is an intelligence aid, not a definitive ranking.
